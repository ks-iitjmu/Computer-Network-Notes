Chapter 1: Computer Networks and the Internet

1.1 What Is the Internet?

The Nuts-and-Bolts View

  • Hosts (End Systems): Billions of connected devices (PCs, servers, IoT, smartphones). Run network apps at network "edge".
  • Communication Links: Fiber, copper, radio, satellite. Performance metric: Transmission Rate (bandwidth, measured in bits/sec).
  • Packet Switches: Forward packets (chunks of data).
    • Routers: Typically in the network core.
    • Link-layer Switches: Typically in access networks.
  • Path/Route: Sequence of communication links and packet switches traversed by a packet from sender to receiver.
  • ISP (Internet Service Provider): Network of packet switches/links. Lower-tier ISPs (local) connect to Upper-tier ISPs (National/Global). Each ISP is independently managed, runs the IP protocol.
  • Protocols: Control sending/receiving of messages. TCP (Transmission Control Protocol) & IP (Internet Protocol) - most important protocols.
  • Standards:
    • IETF: Internet Engineering Task Force (develops Internet standards).
    • RFCs: Requests for Comments (technical docs defining protocols like TCP, IP, HTTP, SMTP).
    • IEEE 802: LAN/WiFi standards (802.3 Ethernet, 802.11 WiFi).

The Services View

  • Infrastructure: Provides services to distributed applications (Web, VoIP, Games, Email, Social Networks).
  • Distributed Applications: Apps involving multiple end systems that exchange data (e.g., Web browser & server).
  • Socket Interface: API provided by End Systems. Set of rules allowing a program to ask the Internet infrastructure to deliver data. Analogous to postal service "envelope/addressing" rules.
    • Specifies how sending program must "package" data.
    • Infrastructure delivers to destination program's socket.
  • Internet provides multiple services: Different services differ in timing/bandwidth guarantees.

Protocols

  • Definition: Defines the format and order of messages exchanged between two or more communicating entities, and the actions taken on transmission/receipt.
  • Human Analogy: "Hi" -> "Hi" -> "Time?" -> "2:00". Specific messages sent, specific actions taken when messages received.
  • Machine Protocol: TCP Connection Request -> TCP Connection Reply -> GET -> File.
  • Key Elements of a Protocol:
    • Syntax: Format/structure of data (how bits are arranged).
    • Semantics: Meaning of each section of bits.
    • Timing: When data should be sent and how fast.
  • Protocol Stack: Protocols are organized in layers, each layer using services of layer below.

1.2 The Network Edge

  • Hosts: Clients (desktop, mobile, laptop) & Servers (data centers).
  • Data Centers: Massive server farms (Google has ~19 major ones, 100k+ servers each). Interconnected by high-speed internal network.
  • Client: Requests and receives service from servers.
  • Server: Always-on host that services requests from clients.

End System Functions

  • Sending Side: Breaks application messages into smaller chunks (packets) of length L bits.
  • Transmits packet into access network at transmission rate R (link capacity/bandwidth).
  • Packet Transmission Delay = L (bits) / R (bits/sec)

Access Networks

Network that physically connects an end system to the first router ("edge router") on a path from end system to any other distant end system.

1. DSL (Digital Subscriber Line)
  • Medium: Existing telephone line (Twisted Pair Copper) to central office DSLAM.
  • Device: DSL Modem (home) <-> DSLAM (Digital Subscriber Line Access Multiplexer at Central Office).
  • Multiplexing (FDM):
    • 0-4 kHz: Regular Telephone (voice).
    • 4-50 kHz: Upstream Data (medium speed).
    • 50 kHz - 1 MHz: Downstream Data (high speed).
  • Splitter: At customer premises, separates voice and data signals.
  • Speed: Standards vary (e.g., ADSL: 24 Mbps down / 2.5 Mbps up). Asymmetric by design (users download more than upload).
  • Architecture: Dedicated line to CO. Performance not affected by neighbors.
  • Distance Limitation: Performance degrades significantly beyond 5-10 miles from CO.
2. Cable Internet
  • Medium: Existing Cable TV infrastructure. HFC (Hybrid Fiber Coax): Fiber to neighborhood node (fiber node), Coax to home.
  • Device: Cable Modem <-> CMTS (Cable Modem Termination System at Head End).
  • Standard: DOCSIS (Data Over Cable Service Interface Specification).
  • Architecture: Shared Broadcast Medium. Every packet sent by head end travels downstream to all homes. Users in a neighborhood share the downstream/upstream capacity. Throughput drops if neighbors are active.
  • Speed: Can be higher than DSL (e.g., DOCSIS 3.1: 10 Gbps down / 1 Gbps up). Asymmetric design.
  • Collision Domain: Upstream requires distributed multiple access protocol to coordinate transmissions.
3. Fiber To The Home (FTTH)
  • Medium: Optical Fiber directly from CO to home.
  • Devices: ONT (Optical Network Terminator) at home <-> OLT (Optical Line Terminator) at CO.
  • Types:
    • AON (Active Optical Network): Switched Ethernet (requires powered equipment in field). Each user has dedicated fiber.
    • PON (Passive Optical Network): Uses unpowered optical splitters (no power needed). Shares fiber from OLT to splitter. Used by Verizon FiOS, AT&T Fiber.
  • Speed: Gbps range (symmetric). Most promising technology.
  • Deployment Cost: Higher than DSL/Cable but declining.
4. Dial-Up (Legacy)
  • Medium: Telephone line (same line as voice, cannot use simultaneously).
  • Speed: Up to 56 kbps (extremely slow by modern standards).
  • Usage: Largely obsolete, but still exists in remote areas.
5. Satellite Internet
  • Medium: Satellite dish at home to satellite in orbit.
  • Speed: Mbps range (improving with LEO constellations like Starlink).
  • Latency: High for GEO satellites (~600ms RTT), lower for LEO (~20-40ms).
  • Use Case: Rural/remote areas without other infrastructure.

Enterprise & Home Networks

  • Ethernet: Wired access. Twisted pair to Ethernet switch. Speeds: 100 Mbps / 1 Gbps / 10 Gbps / 100 Gbps.
  • Home Network Components:
    • Cable/DSL modem
    • Router/Firewall/NAT
    • Wireless Access Point (often combined as "home router")
    • Wired Ethernet connections

Wireless Access Networks

  • WiFi (WLAN): IEEE 802.11. Wireless users transmit to Access Point (AP) connected to wired network.
    • 802.11b/g/n: 2.4 GHz band, various speeds up to 600 Mbps.
    • 802.11ac (WiFi 5): 5 GHz band, up to 3.5 Gbps.
    • 802.11ax (WiFi 6): 2.4/5/6 GHz, up to 10 Gbps.
    • Range: ~10-100 meters.
  • Wide-Area Wireless (Cellular):
    • 3G: ~1 Mbps.
    • 4G LTE: ~10-100 Mbps.
    • 5G: ~100 Mbps - 10 Gbps. Ultra-low latency (~1ms).
  • 5G Fixed Wireless Access: Beam-forming from 5G base station to home modem. Competitor to DSL/Cable/Fiber.
  • Key Difference: WiFi = short range, infrastructure-based. Cellular = wide range, provider-managed.

Physical Media

Bit: Travels by propagating electromagnetic waves or optical pulses across a physical medium.

  • Guided Media (Solid Medium): Waves guided along a solid medium.
    • Twisted Pair (TP): Two insulated copper wires twisted together (reduces EMI). Cheap, ubiquitous.
      • UTP (Unshielded Twisted Pair) - Cat5e/6/6a.
      • Rate depends on wire thickness, distance to switch, modulation.
      • Cat6a: 10 Gbps for 100m.
    • Coaxial Cable: Two concentric copper conductors. Better shielding than TP. Shared medium capability (cable TV). High bandwidth over long distances.
    • Fiber Optics: Thin glass fibers carrying light pulses (each pulse = 1 bit).
      • Ultra-low attenuation (signal loss): ~0.2 dB/km.
      • Immune to electromagnetic interference.
      • Very hard to tap (secure).
      • Long-haul backbone: 10s-100s Gbps per fiber.
      • Cost: Higher than copper (fiber itself cheap, optical transmitters/receivers expensive).
  • Unguided Media (Atmosphere/Space): Waves propagate in atmosphere/space.
    • Terrestrial Radio Channels:
      • Path loss: Signal attenuates as it travels.
      • Shadow fading: Signal blocked by obstacles (buildings, mountains).
      • Multipath fading: Signal reflects off objects, causing interference.
      • Interference: Other signals in same frequency band.
    • Satellite Channels:
      • Geostationary (GEO): 36,000 km altitude. Stationary relative to earth. 280 ms one-way propagation delay (~560ms RTT).
      • MEO (Medium Earth Orbit): ~5,000-12,000 km. GPS satellites.
      • LEO (Low Earth Orbit): ~500-2000 km. Low delay (~20-40ms RTT). Satellites move relative to earth (constellations needed, e.g., Starlink, OneWeb).

1.3 The Network Core

The mesh of packet switches and links interconnecting the Internet's end systems.

Packet Switching

  • Concept: Application-layer messages broken into Packets (L bits). Each packet travels independently through network links and switches.
  • Store-and-Forward Transmission: Switch must receive entire packet before it can begin transmitting the first bit of the packet to the next link.
    • Packet must be completely stored before forwarding.
    • This is why we get per-hop delays.
  • Transmission Delay for 1 packet on 1 link = L / R
  • End-to-End Delay (N links, no prop/proc/queue) = N * (L/R)
  • Example: 1 packet, 3 links, L = 1000 bits, R = 1 Mbps:
    Delay = 3 × (1000/10⁶) = 3 ms
  • Queuing Delay and Packet Loss:
    • Each packet switch has multiple links attached.
    • For each attached link, packet switch has an Output Buffer (Output Queue).
    • If arrival rate > output link transmission rate, packets queue in buffer.
    • Packet Loss: If buffer is full (finite capacity), arriving packets are dropped.
  • Forwarding Table: Each router has a table mapping destination IP addresses (or portions thereof) to outbound links.
  • Routing Protocols: Algorithms that automatically set the forwarding tables.
    • Interior: OSPF, RIP
    • Exterior: BGP
  • Self-Routing: Each packet contains destination address in header. Router examines address, consults forwarding table, forwards to appropriate outbound link.

Circuit Switching

  • Concept: Dedicated end-to-end path between source and destination. Resources (bandwidth/buffers) reserved for entire duration of session.
  • Call Setup: Path must be established before communication begins.
  • Analogy: Restaurant requiring reservation (Circuit) vs Walk-in and hope for table (Packet).
  • Pros:
    • Guaranteed constant transmission rate.
    • No variable queuing delays.
    • Predictable performance.
  • Cons:
    • Inefficient during "silent periods" (resources wasted even when not transmitting).
    • Call setup/tear-down overhead.
    • Dedicated resources cannot be shared.
  • Multiplexing in Circuit-Switched Networks:
    • FDM (Frequency-Division Multiplexing): Link's frequency spectrum divided among connections. Each circuit gets dedicated frequency band. Width of band = bandwidth.
      Example: FM radio stations in different frequencies.
    • TDM (Time-Division Multiplexing): Time divided into frames, frames into slots. Each circuit gets dedicated time slot(s) in each frame.
      Transmission Rate = (Bits per Slot) × (Frame Rate)
      Example: 8000 frames/sec, 8 bits/slot = 64 kbps per circuit.

Packet vs Circuit Switching (Numerical Comparison)

Scenario: 1 Mbps link. Each user needs 100 kbps when active. Each user is active only 10% of the time.
  • Circuit Switching:
    Max users = 1 Mbps / 100 kbps = 10 users (dedicated allocation).
  • Packet Switching:
    With 35 users, probability of >10 users being active simultaneously:
    P(>10 active) = Σ C(35,k) × 0.1^k × 0.9^(35-k) for k=11 to 35 ≈ 0.0004
    Can support 35 users with negligible probability of overload.
  • Conclusion: Statistical Multiplexing Gain - Packet switching wins for bursty data!

Why Packet Switching?

  • Better sharing of transmission capacity: Resources used on demand rather than reserved.
  • Simpler, no call setup.
  • Better for bursty traffic: Data transmission tends to be bursty (bursts of activity with idle periods).
  • Cons of Packet Switching:
    • Variable, unpredictable end-to-end delay (not ideal for real-time services like audio/video).
    • Possible packet loss due to buffer overflow.
    • Requires protocols for reliable data transfer (TCP), congestion control.
  • For real-time guarantees: Need additional mechanisms (QoS guarantees - still an unsolved problem for public Internet).

Network of Networks (Internet Structure Evolution)

  • Problem: How to connect billions of access ISPs together?
  • Structure 1: Access ISPs -> Single Global Transit ISP (Monopoly, unrealistic).
  • Structure 2: Access ISPs -> Multiple competing Global Transit ISPs. Transit ISPs must interconnect.
  • Structure 3: Multi-tier hierarchy. Access ISPs -> Regional ISPs -> Tier-1 ISPs (backbone).
  • Structure 4 (Modern Features):
    • PoP (Point of Presence): Group of routers in provider's network where customer ISPs can connect.
    • Multi-homing: ISP connects to 2+ provider ISPs (reliability, cost optimization).
    • Peering: Direct traffic exchange between same-tier ISPs (bypasses payments to upstream provider). Usually settlement-free.
    • IXP (Internet Exchange Point): Physical location/building where multiple ISPs can peer together. Reduces costs, improves performance.
  • Structure 5 (Today's Internet): Adds Content Provider Networks (Google, Microsoft, Netflix, Meta).
    • Built private global backbones to reduce payments to Tier-1 ISPs.
    • Connect directly (peer) with lower-tier ISPs.
    • Provide better control over user experience.
    • Deploy servers close to users (CDNs).

Tier-1 ISPs

  • Global backbone providers: AT&T, Verizon, Sprint, NTT, Level3/Lumen, etc.
  • Peer with each other (settlement-free) - "Tier-1 club".
  • Do not pay for transit (only receive payments from lower tiers).
  • Have global reach through their own infrastructure.

1.4 Delay, Loss, and Throughput in Packet-Switched Networks

Performance metrics critical for understanding network behavior.

Overview of Delay

  • As a packet travels from one node to the next, it suffers from several types of delays.
  • These delays accumulate to give the total nodal delay.

Nodal Delay Components

dnodal = dproc + dqueue + dtrans + dprop

  1. Processing Delay (dproc):
    • Time to examine packet header and determine where to direct packet.
    • Check for bit-level errors.
    • Typically microseconds or less in high-speed routers.
    • Depends on router processing speed.
  2. Queuing Delay (dqueue):
    • Time waiting in output buffer queue for transmission onto link.
    • Depends on congestion level (number of earlier-arriving packets in queue).
    • Can range from microseconds to milliseconds in practice.
    • Most interesting and complex component.
  3. Transmission Delay (dtrans):
    • Time to push all packet's bits onto the link ("emit the packet").
    • dtrans = L / R
    • L = packet length (bits), R = link transmission rate (bps).
    • Depends on packet size and link bandwidth.
    • Example: L = 1000 bits, R = 1 Mbps → dtrans = 1 ms.
  4. Propagation Delay (dprop):
    • Time for a bit to propagate from start of link to next router.
    • dprop = d / s
    • d = length of physical link (meters), s = propagation speed in medium.
    • s ≈ 2×10⁸ to 3×10⁸ m/s (speed of light in medium).
    • Depends on physical medium and distance.
    • Example: d = 1000 km, s = 2×10⁸ m/s → dprop = 5 ms.

Transmission vs Propagation: Key Difference

  • Transmission Delay: Function of packet size (L) and link rate (R). Time to push bits onto link. Has NOTHING to do with distance.
  • Propagation Delay: Function of distance (d) and speed (s). Time for bit to travel. Has NOTHING to do with packet size.
  • Common confusion: Students often confuse these! Remember:
    • Transmission = "Time to send" (like loading cars onto highway)
    • Propagation = "Time to travel" (like car driving on highway)

Caravan Analogy (Transmission vs Propagation)

  • Scenario: 10 cars (bits in packet) in a caravan. Toll booth (Router). 100 km highway segment.
  • Cars travel at 100 km/h (Propagation Speed).
  • Toll booth takes 12s to service each car (Transmission Rate analog).
  • Transmission Delay: Time for whole caravan to pass through booth = 12s × 10 cars = 120s = 2 min.
  • Propagation Delay: Time for one car to drive 100km = 100km / 100km/h = 1 hour.
  • End-to-End: First car arrives at next toll booth after 60 min. Last car of caravan arrives after 60 min + 2 min = 62 min.

Queuing Delay and Traffic Intensity

  • Traffic Intensity: Key metric for understanding queuing behavior.
    Traffic Intensity I = (L × a) / R
  • L = average packet size (bits), a = average packet arrival rate (packets/sec), R = link transmission rate (bps).
  • La = average bits arriving per second. R = bits that can be transmitted per second.
Traffic Intensity Analysis:
  • I ≈ 0: Packets arrive far apart. Average queuing delay is small.
  • I → 1: Arrival rate approaches capacity. Average queuing delay grows exponentially.
  • I > 1: More bits arriving than can be transmitted. Queue grows without bound → Infinite delay (packets will be lost).
  • Golden Rule: Design systems so that traffic intensity < 1.

Packet Loss

  • Cause: Queues (buffers) in routers have finite capacity.
  • If packet arrives to find a full queue, router drops the packet (packet is lost).
  • Lost packet may be retransmitted by previous node, by source end system, or may not be retransmitted at all (depends on protocol).
  • Packet Loss Rate: Fraction of packets lost. Increases as traffic intensity approaches 1.
  • Performance trade-off: Larger buffers → less loss but more delay. Smaller buffers → less delay but more loss.

End-to-End Delay

  • For N-1 routers (N links) between source and destination, assuming identical delays at each node:
    dend-end = N × (dproc + dqueue + dtrans + dprop)
  • If we neglect queuing delays (unloaded network):
    dend-end = N × (dproc + dtrans + dprop)

Traceroute and Round-Trip Time (RTT)

  • Traceroute: Diagnostic program to measure delay along the path.
  • How it works:
    • Source sends packets with TTL (Time to Live) = 1, 2, 3, ... N.
    • When TTL reaches 0 at router n, router discards packet and sends ICMP "Time Exceeded" message back to source.
    • Source measures Round-Trip Time (RTT) for each router.
    • Sends 3 packets per TTL value, reports 3 RTT measurements.
  • Output interpretation:
    • Shows each hop along the path.
    • * indicates lost probe (no response).
    • RTT variations show queuing delay variability.
  • RTT vs One-Way Delay: RTT ≈ 2 × One-way delay (approximately, if paths are symmetric).

Throughput

  • Definition: Rate (bits/time unit) at which bits are being transferred between sender and receiver.
  • Instantaneous Throughput: Rate at a given point in time.
  • Average Throughput: Rate over longer period. If F bits transferred in T seconds:
    Average Throughput = F / T bits/sec
  • Bottleneck Link: Link on end-to-end path that constrains end-to-end throughput.
    Throughput = min(R₁, R₂, ..., Rₙ) = min(Rs, Rc)
    where Rs = server access link, Rc = client access link.
Throughput Example - Single Connection:
  • Server connects at Rs bps. Client connects at Rc bps. Core network rate R (very high).
  • If R >> Rs and R >> Rc: Throughput ≈ min(Rs, Rc)
  • In modern Internet, core is fast. Bottleneck is usually the access network.
Throughput Example - Multiple Connections:
  • 10 connections sharing a link R. Each gets R/10 average.
  • Bottleneck may shift to shared link in core:
    Throughput per connection = min(Rs, Rc, R/10)

Delay vs Throughput

  • Different metrics: Low delay ≠ high throughput.
  • Example: Satellite link can have high throughput (many bits per second) but high delay (280ms propagation).
  • Fiber can have low delay AND high throughput.
  • Both matter for different applications:
    • VoIP, gaming: Need low delay.
    • File download: Need high throughput.
    • Video streaming: Need both!

1.5 Protocol Layers and Their Service Models

Why Layering?

  • Dealing with complex systems: Networks are complex! Layering provides a structured way to discuss system components.
  • Modularization: Each layer provides a service to the layer above, using services of layer below.
  • Advantages:
    • Easier identification and relationship of components.
    • Easier maintenance/updating (change in one layer doesn't affect others if interface remains same).
    • Explicit structure allows easy understanding of complex systems.
  • Disadvantages:
    • Header overhead at each layer.
    • Possible duplication of functionality (e.g., error checking in multiple layers).
    • Layering may hide information needed for optimization.

Service Models

  • Service: Set of operations a layer provides to the layer above it.
  • Service Model: Defines what guarantees the layer provides.
    • Connection-oriented: Establish connection before data transfer (setup phase).
    • Connectionless: No setup, just send data.
    • Reliable: Guarantees delivery (may use acknowledgments, retransmissions).
    • Unreliable: Best-effort, no guarantee.
  • Protocol: Defines the format and order of messages exchanged between peer entities (same layer on different hosts).

The Internet Protocol Stack (5 Layers)

1. Application Layer
  • Function: Where network applications and their application-layer protocols reside.
  • Responsibilities: Creating user data, defining message semantics.
  • Protocols: HTTP (Web), SMTP (email), FTP (file transfer), DNS (name resolution), DHCP (address allocation).
  • PDU (Protocol Data Unit): Message.
  • Examples: Web browser, email client, streaming apps.
2. Transport Layer
  • Function: Process-to-process data transfer. Transports application-layer messages between application endpoints.
  • Multiplexing/Demultiplexing: Uses port numbers to deliver data to correct process.
  • Protocols:
    • TCP (Transmission Control Protocol):
      • Connection-oriented (3-way handshake).
      • Reliable data transfer (acknowledgments, retransmissions).
      • Flow control (prevent overwhelming receiver).
      • Congestion control (prevent overwhelming network).
    • UDP (User Datagram Protocol):
      • Connectionless ("fire and forget").
      • Unreliable - no guarantees.
      • No flow/congestion control.
      • Fast, lightweight, used for streaming, DNS, gaming.
  • PDU: Segment (TCP) or Datagram (UDP).
3. Network Layer
  • Function: Moving packets from source host to destination host (host-to-host delivery).
  • Responsibilities:
    • Forwarding: Moving packets from router's input to appropriate output (local action).
    • Routing: Determining the path packets take from source to destination (global action).
  • Key Protocol: IP (Internet Protocol)
    • Defines datagram format.
    • Addressing conventions (IP addresses).
    • Packet handling conventions.
    • Best-effort service: No guarantees on delivery, delay, or order.
  • Routing Protocols: OSPF, RIP (interior), BGP (exterior).
  • PDU: Datagram.
  • Devices: Routers.
4. Link Layer
  • Function: Moving frames from one node to neighboring node (node-to-node delivery over a single link).
  • Responsibilities:
    • Framing (encapsulating datagram into frame).
    • Link access (MAC protocol for shared medium).
    • Error detection/correction.
    • Reliable delivery between adjacent nodes (optional).
  • Protocols: Ethernet (802.3), WiFi (802.11), PPP, HDLC.
  • PDU: Frame.
  • Addressing: MAC addresses (48-bit hardware address).
  • Devices: Switches, NICs (Network Interface Cards).
  • Note: Different links may use different protocols (e.g., Ethernet at home, WiFi at coffee shop).
5. Physical Layer
  • Function: Moving individual bits "on the wire" from one node to the next.
  • Responsibilities:
    • Bit representation (voltages, light pulses).
    • Transmission rate.
    • Physical characteristics of transmission media.
    • Synchronization of bits.
  • Protocols: Dependent on medium (e.g., DSL PHY, Ethernet PHY, 802.11 PHY).
  • Concerned with: Mechanical, electrical, timing specifications.

OSI Reference Model (7 Layers)

  • OSI (Open Systems Interconnection) model has 7 layers.
  • Internet stack (5 layers) is missing two layers present in OSI:
  • 6. Presentation Layer:
    • Data representation and encryption.
    • Compression, data conversion.
    • Allow applications to interpret meaning of data.
  • 5. Session Layer:
    • Synchronization, checkpointing, recovery of data exchange.
    • Establishing, managing, terminating sessions.
  • Internet philosophy: These services (if needed) are implemented in the Application Layer. Keep complexity at the edges (End-to-End Principle).
  • OSI Full Stack: Application → Presentation → Session → Transport → Network → Data Link → Physical.

Encapsulation

  • Concept: Each layer takes data from above, adds header information, creates new data unit.
  • At sending host:
    • Application: Creates Message (M).
    • Transport: Adds Transport Header (Ht) → Segment (Ht + M).
    • Network: Adds Network Header (Hn) → Datagram (Hn + Ht + M).
    • Link: Adds Link Header (Hl) → Frame (Hl + Hn + Ht + M).
    • Physical: Transmits bits.
  • At receiving host: Reverse process (decapsulation). Each layer removes its header and passes data up.

What Each Device Processes

  • Link-Layer Switch: Processes up to Layer 2 (Link). Examines frame header, makes forwarding decision based on MAC address.
  • Router: Processes up to Layer 3 (Network). Examines datagram header, makes routing decision based on IP address.
  • Host: Processes all 5 layers. Full protocol stack implementation.
  • Firewall/Middleboxes: May process up to Layer 7 (Application) for deep packet inspection.

End-to-End Principle

  • Key design principle of the Internet.
  • Idea: Implement functionality at the endpoints (hosts), not in the network core.
  • Why?
    • Keeps network core simple, fast, and scalable.
    • Only endpoints know true requirements.
    • Network can focus on basic delivery.
  • Examples:
    • Reliable data transfer: Implemented at transport layer (TCP), not in routers.
    • Error checking: End systems verify, not every intermediate node.

1.6 Networks Under Attack (Security)

Internet was originally designed as "a group of mutually trusting users attached to a transparent network." Security was an afterthought.

The Bad Guys Can Put Malware into Your Host

  • Malware: Malicious software that can infect hosts via the Internet.
  • Types of Malware:
    • Virus:
      • Requires user interaction to infect (e.g., opening email attachment, downloading file).
      • Self-replicating: copies itself to other programs/files.
    • Worm:
      • Self-replicating, spreads automatically without user interaction.
      • Exploits vulnerabilities in networked software.
      • Can spread very rapidly across Internet.
    • Trojan Horse:
      • Disguised as legitimate software.
      • Provides backdoor access or performs malicious actions.
    • Ransomware:
      • Encrypts victim's files.
      • Demands payment for decryption key.
    • Spyware:
      • Monitors user activity, collects sensitive information.
      • Keyloggers capture keystrokes (passwords, credit cards).
  • Botnet:
    • Network of compromised devices ("zombies" or "bots").
    • Controlled by attacker (botmaster).
    • Used for spam, DDoS attacks, cryptocurrency mining.
    • Can consist of millions of infected computers.

The Bad Guys Can Attack Servers and Network Infrastructure

Denial-of-Service (DoS) Attacks:
  • Goal: Make a resource (server, bandwidth) unavailable to legitimate users.
  • Types:
    • Vulnerability Attack:
      • Send specially crafted malformed packets.
      • Exploits software vulnerability to crash or hang the host.
    • Bandwidth Flooding:
      • Send flood of packets to saturate target's access link.
      • Legitimate packets cannot get through.
    • Connection Flooding:
      • Establish many half-open or fully open TCP connections.
      • SYN Flood: Send many TCP SYN requests, never complete handshake.
      • Exhausts server's resources for tracking connections.
Distributed Denial-of-Service (DDoS):
  • Attack launched from multiple sources (usually a botnet).
  • Much harder to defend against than single-source DoS.
  • Difficult to filter because traffic comes from many IP addresses.
  • Attack traffic can be larger than target's total bandwidth.

The Bad Guys Can Sniff Packets

  • Packet Sniffer: Passive receiver that records a copy of every packet that passes by.
  • How it works:
    • Place receiver near sender (same WiFi network, same broadcast LAN).
    • Set network interface to promiscuous mode (receives all packets, not just those addressed to it).
    • Software like Wireshark records and analyzes packets.
  • Danger: Can capture passwords, emails, sensitive data if unencrypted.
  • Defense: Encryption (HTTPS, TLS, VPN). Even if sniffed, data is unreadable.
  • Common locations: Coffee shop WiFi, corporate LAN, Internet backbone (nation-state surveillance).

The Bad Guys Can Masquerade as Someone You Trust

IP Spoofing:
  • Inject packets into Internet with a false source IP address.
  • Receiver believes packet came from trusted source.
  • Used in DoS attacks, to hide identity, or to bypass IP-based access controls.
  • Defense: End-point authentication (cryptographic verification of identity).
Man-in-the-Middle (MITM) Attack:
  • Attacker secretly relays and possibly alters communication between two parties.
  • Both parties believe they are communicating directly with each other.
  • Attacker can eavesdrop, modify messages, inject false data.
  • Defense: Mutual authentication, encryption (TLS with certificate verification).
DNS Spoofing/Poisoning:
  • Corrupt DNS cache to redirect users to malicious sites.
  • User types legitimate URL but goes to attacker's server.
  • Defense: DNSSEC (cryptographic authentication of DNS responses).

Security Principles (Defense)

  • Confidentiality: Only sender and intended receiver should understand message contents (encryption).
  • Integrity: Sender/receiver should be able to detect message alteration (message digests, digital signatures).
  • Authentication: Sender and receiver should be able to confirm identity of other party.
  • Access Control: Services must be accessible only to authorized users.
  • Availability: Services must be accessible when needed (counter DoS attacks).

Lines of Defense

  • Firewalls: Filter packets at network edge based on rules.
  • Intrusion Detection Systems (IDS): Monitor traffic for suspicious activity, alert administrators.
  • Intrusion Prevention Systems (IPS): Actively block detected attacks.
  • Encryption: Protect data confidentiality and integrity.
  • Authentication mechanisms: Verify identities (passwords, certificates, multi-factor).
  • Regular patching: Fix vulnerabilities in software.

1.7 History of Computer Networking and the Internet

1961-1972: Development of Packet Switching

  • 1961 - Kleinrock (MIT):
    • First paper on queuing theory applied to networks.
    • Demonstrated effectiveness of packet switching for bursty traffic.
    • Analytical foundation for computer networking.
  • 1964 - Paul Baran (RAND):
    • Packet switching for secure military voice communications.
    • Proposed distributed network architecture (no single point of failure).
  • 1967 - ARPANET planned:
    • Funded by ARPA (Advanced Research Projects Agency, US DoD).
    • Goal: Connect research computers at universities.
  • 1969 - First ARPAnet node:
    • Installed at UCLA (Kleinrock's lab).
    • First message sent to Stanford Research Institute (SRI).
    • Intended: "LOGIN" | Sent: "LO" (system crashed after 2 letters!).
  • 1970: ALOHAnet (Hawaii) - first wireless packet-switched network (Norman Abramson).
  • 1972:
    • ARPAnet public demonstration (15 nodes).
    • First email program (Ray Tomlinson) - invented @ symbol for email addresses.
    • NCP (Network Control Protocol) - first host-to-host protocol.

1972-1980: Proprietary Networks and Internetworking

  • 1973 - Metcalfe's Ph.D. thesis: Ethernet (inspired by ALOHAnet). Published 1976.
  • 1974 - Cerf and Kahn:
    • Published architecture for interconnecting networks.
    • Foundation for TCP/IP.
    • Key principles: Minimalist network core, intelligence at edges.
  • Proprietary architectures: IBM SNA, DEC DECnet, Xerox XNS.
  • 1976: First Ethernet at Xerox PARC.
  • Late 1970s:
    • Fixed-length packet switching (ATM precursor).
    • ARPANET had ~200 hosts.

1980-1990: Protocol Proliferation

  • January 1, 1983 - "Flag Day":
    • ARPANET switches from NCP to TCP/IP.
    • All hosts required to migrate simultaneously.
    • Birth of the modern Internet.
  • 1983: DNS (Domain Name System) defined.
  • 1985: NSFnet created by National Science Foundation.
    • Connected university supercomputer centers.
    • Initially 56 kbps backbone, later upgraded.
  • Mid-1980s:
    • TCP congestion control added (Van Jacobson).
    • Addressed "congestion collapse" problem.
  • Minitel (France, 1984):
    • Pre-Web online service, great success in France.
    • Had email, online banking, info services before WWW.
    • Government-deployed terminals in households.
  • By 1990: ARPANET decommissioned. ~100,000 hosts on Internet.

1990s: The World Wide Web and Explosion

  • 1989-1991 - Tim Berners-Lee (CERN):
    • Invented the World Wide Web.
    • Created: HTTP (HyperText Transfer Protocol).
    • Created: HTML (HyperText Markup Language).
    • Created: First browser and server.
    • Original purpose: Share research documents.
  • 1993: Mosaic browser (NCSA, Marc Andreessen) - first popular graphical browser.
  • 1994: Netscape founded. Browser wars begin.
  • 1995:
    • NSFnet backbone decommissioned - commercial backbone takes over.
    • Internet becomes commercial.
    • ~50 million hosts.
  • Killer Applications of 1990s:
    • Email (already existed but exploded in popularity).
    • Web browsing.
    • Instant messaging (ICQ, AIM).
    • P2P file sharing (Napster, 1999).
  • Late 1990s: Dot-com boom. Internet startups proliferate.

2000s-Present: The New Millennium

  • 2000: Dot-com bust. Many Internet companies fail.
  • Broadband explosion:
    • DSL, Cable modem replace dial-up.
    • Always-on connectivity becomes norm.
  • 2003-2004: Social networking (MySpace, then Facebook 2004).
  • 2005: YouTube - user-generated video explosion.
  • 2007: iPhone - mobile Internet goes mainstream.
  • 2010s:
    • Cloud computing (AWS, Azure, Google Cloud).
    • 4G LTE widespread.
    • Streaming services (Netflix, Spotify).
    • Internet of Things (IoT) - billions of connected devices.
    • Video constitutes majority of Internet traffic.
  • 2020s:
    • 5G deployment.
    • LEO satellite constellations (Starlink).
    • ~5+ billion Internet users worldwide.
    • AI-generated content.

Key Figures in Internet History

  • Leonard Kleinrock: Queuing theory for packet switching.
  • Paul Baran: Distributed packet-switched networks.
  • Vint Cerf & Bob Kahn: TCP/IP architecture ("Fathers of the Internet").
  • Ray Tomlinson: Email, @ symbol.
  • Tim Berners-Lee: World Wide Web, HTTP, HTML.
  • Marc Andreessen: Mosaic browser, Netscape.
  • Bob Metcalfe: Ethernet.
  • Van Jacobson: TCP congestion control.

Key Formulas Summary

  • Transmission Delay: dtrans = L / R (L=bits, R=bps)
  • Propagation Delay: dprop = d / s (d=distance, s=speed)
  • Nodal Delay: dnodal = dproc + dqueue + dtrans + dprop
  • End-to-End Delay (N links): de2e = N × (dproc + dtrans + dprop)
  • Traffic Intensity: I = La / R (L=pkt size, a=arrival rate)
  • Throughput: = min(Rs, Rc, Rlink)
  • TDM Rate: = (bits/slot) × (frames/sec)
  • Circuit Capacity: = Total Bandwidth / Bandwidth per user

Important Definitions

  • Protocol: Rules defining format, order of messages, and actions on transmission/receipt.
  • Packet: Chunk of data with header information.
  • Router: Network-layer packet switch (forwards based on IP).
  • Switch: Link-layer packet switch (forwards based on MAC).
  • ISP: Internet Service Provider - network of routers and links.
  • RFC: Request for Comments - Internet standards documents.
  • Socket: Interface between application and transport layer.
  • Bandwidth: Maximum rate at which data can be transferred.
  • Throughput: Actual rate of successful data transfer.
  • Latency/Delay: Time for packet to travel from source to destination.
  • RTT: Round-Trip Time - time for packet to go and acknowledgment to return.
